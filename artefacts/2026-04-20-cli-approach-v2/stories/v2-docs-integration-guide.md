## Story: Operator integration guide — first-run experience and workflow-declaration authoring runbook

**Epic reference:** `../epics/e4-operator-docs.md`
**Discovery reference:** `../discovery.md`
**Benefit-metric reference:** `../benefit-metric.md`

## User Story

As a **platform operator adopting the CLI on a new project**,
I want to **a single integration guide covering install, first cycle, upgrade, and workflow-declaration authoring**,
So that **adoption on a new project is reproducible without session-specific context or engineering assistance — I can re-run the install path in six months without re-deriving it from source reading**.

## Benefit Linkage

**Metric moved:** M5 (adoption reproducibility), MM1 (adoption friction documented, thus traceable).
**How:** The tool being adoptable is not the same as the tool being documented. Even for self-consumption, a guide eliminates "I did it once but can't remember the steps" regressions. For post-MVP second-party adoption, the guide is the hand-off surface.

## Architecture Constraints

- **C7 one question at a time** — guide prose follows the conversational-pattern convention when documenting CLI-mediated skill invocations.
- **No embedded design content** (`product/constraints.md` §8) — guide references external platform / design docs by URL, not embedded copies.
- **MC-SEC-02** — no credentials in documentation (obvious, noted for completeness).

## Dependencies

- **Upstream:** Epic 3 complete (v2-e2e-project1, v2-e2e-project2, v2-e2e-upgrade DoD) — documented first-run experience is grounded in observed cycles.
- **Upstream:** Epic 1 + Epic 2 DoD — install and declaration flows are the guide's subject matter.
- **Downstream:** None (MVP end-state).

## Acceptance Criteria

**AC1:** Given Epic 3 is DoD with cycles 1 and 2 observed, When the integration guide is authored, Then it contains at least these sections: (a) prerequisites, (b) install steps (collisions pre-flight + sidecar + lockfile), (c) first feature cycle walk-through, (d) upgrade flow, (e) workflow-declaration authoring runbook (for adding a local skill post-MVP), (f) troubleshooting with observed friction points from Epic 3 cycles.

**AC2:** Given the guide is complete, When an operator unfamiliar with this session follows it end-to-end on a hypothetical project 3, Then they can complete install + first cycle without consulting source code or session logs — the guide's worked example is sufficient.

**AC3:** Given the guide includes the workflow-declaration authoring runbook, When an operator uses it to add a single custom node to a project's declaration, Then the runbook's steps produce a valid declaration that passes `skills-repo verify --declaration` without manual schema consultation.

**AC4:** Given friction points from Epic 3 are documented, When the guide's troubleshooting section is inspected, Then each documented friction has: (a) the observed symptom, (b) the diagnosis, (c) the resolution — not just a FAQ-dump but cycle-derived content.

**AC5:** Given the guide is committed to the v2 feature folder, When it is linked from operator-facing README or equivalent entry-point, Then a new operator can locate the guide without prior-session knowledge.

## NFRs

- **Accessibility:** Guide is plain markdown; no dependency on specific rendering. Readable without tool installation.
- **Correctness:** Every CLI command shown in the guide matches the implemented command signature and behaviour; drift triggers a /improve pass.

**Scope stability:** Stable — once Epic 3 observations are in hand, authoring is mechanical.
